# Troubleshooting Failed Edits: `diff` vs `whole` Format

Experiencing a high number of failed edits when using the default `diff` or `editor-diff` formats is a common issue. Switching to the `whole` edit format **could potentially improve the success rate**, especially if the failures relate to how the LLM generates or applies diffs.

**Reasoning:**

1.  **How `diff` Works and Fails:**
    *   Aider sends the current code context to the LLM.
    *   The LLM generates a set of changes, ideally formatted as a diff/patch.
    *   Aider attempts to apply this patch to the local file.
    *   **Potential Failure Points:**
        *   The LLM might generate an invalid or malformed diff.
        *   The context lines in the diff generated by the LLM might not perfectly match the current state of the file (especially if the file changed slightly or the LLM misinterpreted the context).
        *   Complex, overlapping, or numerous changes within a single block can make generating a correct diff harder for the LLM.
        *   In Architect mode with `editor-diff`, the editor LLM might struggle to refine a complex diff from the main LLM.

2.  **How `whole` Works and Helps:**
    *   Aider sends the current code context to the LLM.
    *   The LLM generates the **complete, intended content** of the file(s).
    *   Aider receives this full content and simply replaces the existing file content with the new content provided by the LLM.
    *   **Advantages:**
        *   **Bypasses Patching:** It completely avoids the complexities and potential errors associated with generating and applying diffs.
        *   **Full Context for LLM Output:** The LLM generates based on what it thinks the *entire file* should look like, which can sometimes be easier for it than calculating precise line changes.
        *   **Simpler for Aider:** Aider's task becomes a straightforward file replacement.

**Potential Downsides of `whole`:**

*   **Token Usage:** Sending the entire file content back and forth uses significantly more tokens than sending just a diff, especially for larger files. This can lead to higher API costs and potentially slower response times.
*   **Context Window Limits:** For very large files, the `whole` format might exceed the LLM's context window limits more easily.
*   **Less Focused Changes:** Sometimes, seeing the whole file might tempt the LLM to make broader, unintended changes if not carefully prompted. `diff` naturally focuses it more on specific modifications.

**Recommendation:**

Given a high frequency of failed edits with `diff`, **it is definitely worth experimenting with the `whole` edit format.**

*   Try it in **Code Mode** first for a task where `diff` was failing.
*   If that works better, try it in **Architect Mode**. Remember, setting `--edit-format whole` in Architect mode primarily changes how the *main* LLM interacts. It doesn't directly change the *editor* interaction unless you were using `editor-whole` (which isn't the default). However, providing the main LLM's output as a whole file might indirectly simplify things downstream.

Monitor the results and token usage. If `whole` resolves the frequent failures, the increased token cost might be an acceptable trade-off for reliability. If it doesn't help or the token usage is prohibitive, the issue might lie elsewhere (e.g., model limitations, prompt clarity, file complexity).
